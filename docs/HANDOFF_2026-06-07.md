## 1. Три тихие регрессии (root cause)

### 1.1 ❌→✅ TG_BOT_TOKEN мёртв
- `/home/crimea_parser/.env`: `TG_BOT_TOKEN` для `@Hotel_Lead2_Bot` возвращал `401 Unauthorized` на любой API call.
- **Эффект:** ни один прогон парсера за неделю не смог отправить отчёт/checkpoint в TG-группу. Пользователь не получал никаких уведомлений.
- **Исправлено:** в этой сессии — пользователь дал новый токен, я обновил `.env` с backup `.env.bak.20260607_000717`. `getMe` теперь `ok:true` (id 8252614039, `@Hotel_Lead2_Bot` aka «База_Отель_бот2»).
- **Сервис не перезапускался** — `crimea_parser.service` oneshot, при следующем старте подсосёт новый токен.

### 1.2 ❌ email_finder убивается systemd по 12h timeout
- Файл лога [email_finder.log.1] показывает: за 12 часов обработано ~437 сайтов (с offset 831 до 1268 из 11 888). Скорость ~36 сайтов/час.
- 06.06 03:45:22 systemd послал SIGTERM — Playwright не успел shutdown, EPIPE на pipe к node.js процессу.
- **Эффект:** не вызываются `build_xlsx` → `gdrive_upload` → `telegram_notify` (они в конце `main.py:166-189`). Цепочка обрывается, XLSX отстаёт, отчёт не уходит.
- **Корневая проблема:** 12K сайтов × ~36/час = **~14 суток непрерывной работы**. Timeout=12h фундаментально мал.
- **Временно обойдено:** в этой сессии вручную через `utils.merger.build_master_xlsx()` пересобрал XLSX из всех `result_enriched_*.csv` (за 75 сек), доставил как `sendDocument` в TG.

### 1.3 ❌ GDrive OAuth refresh-token expired
- `/home/crimea_parser/token.json` отвечает `invalid_grant: Token has been expired or revoked`.
- **Эффект:** `utils.gdrive.upload_file` молча возвращает `None` — файлы не попадают в общую папку. Это маскировалось предыдущими двумя проблемами (никто не звал upload в полном пайплайне).
- **Требует ручной reauth:** перегенерировать `token.json` через OAuth flow (нужен браузер на стороне разработчика, см. [HANDOFF_2026-05-31.md §8](HANDOFF_2026-05-31.md#8-доступы)).

## 2. Бот в группе (отдельная проблема, не связана с молчанием)

`parser_admin_bot.service` (`@Hotels_admin_bot`) использует whitelist:
```python
@router.message(F.func(lambda m: not is_admin(m.chat.id)))
```
В ЛС `chat.id == user_id == 1264067528` → проходит. **В группе** `chat.id == -1003781591836` ≠ user_id → бот молча игнорирует команды.

В этой сессии бот **добавлен в группу «База_Отель_Сбор»** (msg 329 подтверждает доступ). Но `/menu` и другие команды в группе всё ещё не работают — нужно решить одним из вариантов:

| # | Что менять | Безопасность |
|---|---|---|
| A | `ADMIN_CHAT_IDS=1264067528,-1003781591836` + рестарт бота | ❌ Любой участник группы получит контроль над парсером |
| B | Заменить `is_admin(m.chat.id)` → `is_admin(m.from_user.id)` в [handlers/menu.py] (5 мест) и [handlers/commands.py:35] (1 место) | ✅ Команды слушаются только владельца, и в ЛС, и в группе |
| C | Не трогать группу, пользоваться меню в ЛС с `@Hotels_admin_bot` | ✅ Уже работает |
